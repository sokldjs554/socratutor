## 1. 공고 분석 요약

### 회사가 요구하는 것

| 항목 | 내용 |
|------|------|
| 회사 | Socra AI (구 뤼이드) — 산타(토익), 베스트웨이, 리얼아카데미 등 AI 에듀테크 |

### 설계 방향 도출

공고의 주요 업무 4가지(Agent 구현 / 백엔드 서빙 / RAG·메모리 / 평가 파이프라인)를 **전부 한 번씩 건드리는** 프로젝트를 1인 인턴 스케일로 만든다. 각 요소는 최소 구현으로 충분하다 — 깊이보다 "네 가지를 잇는 파이프라인을 스스로 설계했다"가 인턴 포트폴리오의 포인트.

## 2. 프로젝트 개요

### 한 줄 소개

정답을 바로 알려주지 않고 **소크라테스식 질문으로 유도하는 영어 학습 AI 튜터**를 만든다. 학습자의 오답을 기억하고, 문법 지식 베이스를 검색(RAG)해 근거 있는 힌트를 주며, 튜터의 응답 품질을 자동 평가 파이프라인으로 검증한다.

### 왜 이 주제인가

- **회사 이름이 곧 컨셉이다.** Socra AI는 소크라테스 문답법에서 온 이름으로, "정답 대신 질문으로 학습자를 성장시킨다"가 회사 철학이다. 그 철학을 그대로 구현한 프로젝트라 자소서·면접에서 연결이 자연스럽다.
- 회사 도메인(영어 교육 — 산타토익)과 일치한다.
- 공고의 주요 업무 4가지를 정확히 커버한다.

### 인턴스러움 캘리브레이션 (중요)

**넣는 것** — 작지만 파이프라인이 완결된 것:

- 동작하는 에이전트 + API 서버 + 최소 UI
- 작은 RAG (문법 노트 30~50개 문서)와 학습자 오답 메모리
- LLM-as-judge 평가 파이프라인 (시나리오 30개, 기준 3개)
- 요청별 토큰·지연시간·비용 로깅 (우대사항의 "비용을 고려한 운영" 어필)
- 프롬프트 버전별 평가 점수 비교 기록 — "개선을 측정했다"는 증거
- 실행 방법과 트러블슈팅 기록이 있는 정직한 README

**안 넣는 것** — 넣는 순간 인턴답지 않아지는 것:

- ❌ 멀티 에이전트 오케스트레이션 (LangGraph 그래프 설계) — "다음 단계" 섹션으로만 언급
- ❌ 모델 파인튜닝, vLLM 셀프호스팅, GPU 서빙
- ❌ TTS·이미지 생성 모델 연동
- ❌ Kubernetes, Redis 캐싱, 오토스케일링
- ❌ 대규모 벡터 DB 튜닝, 임베딩 모델 비교 실험

## 3. 기술 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| 언어 | **Python 3.11+** | AI 분야 표준, 본인 기존 역량 |
| 백엔드 | **FastAPI** | 가볍고 async 지원, LLM 서빙에 사실상 표준 |
| LLM | **Claude API** (`claude-opus-4-8`) | 공식 Python SDK의 tool use로 에이전트 구현 |
| RAG | **ChromaDB** (로컬 임베디드) | 설치 한 줄, 인턴 스케일에 적정 |
| DB | **SQLite** (→ 확장 시 PostgreSQL) | 학습자 메모리·로그 저장. 단순하게 시작 |
| UI | **Streamlit** | 채팅 데모 화면을 하루 만에. 프론트에 시간 쓰지 않음 |
| 테스트 | pytest + LLM-as-judge 스크립트 | |
| CI | GitHub Actions (lint + 유닛 테스트) | |
| 배포 | **Docker + docker-compose**, 마지막에 **AWS EC2** 데모 배포 1회 | 공고 스킬 태그(AWS·Docker) 대응. Kubernetes는 인턴 스케일 초과 — "다음 단계"로만 언급 |

> **비용 참고:** Claude Opus 4.8은 입력 $5 / 출력 $25 (1M 토큰당). 개발 중 반복 실험이나 평가 파이프라인의 채점자(judge)는 Haiku 4.5($1/$5)로 낮춰 비용을 아낄 수 있다 — 이 선택 자체를 README에 "비용을 고려한 모델 선택"으로 기록하면 우대사항 어필이 된다.

## 4. 핵심 기능 명세

| # | 기능 | 설명 |
|---|------|------|
| F1 | 소크라테스식 튜터 챗 | 학습자가 영어 문법/독해 질문 → 정답 직접 노출 대신 단계적 질문·힌트로 유도. 시스템 프롬프트로 페르소나·금지사항(정답 즉시 공개 금지) 정의 |
| F2 | RAG 지식 검색 | 문법 노트(마크다운 30~50개, 직접 작성/정리)를 ChromaDB에 임베딩. 에이전트가 `search_grammar_notes` 툴로 검색해 근거 있는 힌트 생성 |
| F3 | 학습자 메모리 | 오답·약점을 SQLite에 기록(`record_mistake` 툴). 대화 시작 시 최근 오답 요약을 컨텍스트로 주입 → "지난번에 현재완료를 헷갈리셨죠" 같은 개인화 |
| F4 | 퀴즈 출제·채점 | `make_quiz` 툴로 학습자 약점 기반 문제 출제, 답변 채점 후 결과를 메모리에 기록 |
| F5 | 자동 평가 파이프라인 | 시나리오 30개 × 평가 기준 3개를 LLM-as-judge로 채점하는 스크립트. 프롬프트 수정 전후 점수 비교 리포트 생성 |
| F6 | 운영 로그 | 요청별 토큰 사용량·응답 지연시간·추정 비용을 DB에 기록, `/stats` 엔드포인트로 요약 제공 |

## 6. 에이전트 설계

- **구조**: 싱글 에이전트 + 툴 4개. SDK의 tool runner가 "LLM 호출 → 툴 실행 → 결과 반영" 루프를 처리한다. 멀티 에이전트는 의도적으로 배제 (인턴 스케일 초과 — 확장 섹션 참고).
- **시스템 프롬프트 핵심 규칙**:
  1. 정답을 바로 말하지 않는다 — 학습자가 스스로 도달하도록 최대 3단계 질문/힌트
  2. 힌트는 반드시 검색된 문법 노트에 근거한다 (환각 방지)
  3. 학습자가 3번 틀리면 정답과 해설을 공개하고 오답을 기록한다

```python
from anthropic import Anthropic, beta_tool

client = Anthropic()

@beta_tool
def search_grammar_notes(query: str) -> str:
    """문법 지식 베이스에서 관련 문서를 검색한다.

    Args:
        query: 검색 질의 (예: "현재완료와 과거시제 차이")
    """
    return rag.search(query)

# get_mistake_notes / record_mistake / make_quiz 도 같은 방식으로 정의

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",
    max_tokens=2048,
    system=TUTOR_SYSTEM_PROMPT,
    tools=[search_grammar_notes, get_mistake_notes, record_mistake, make_quiz],
    messages=history + [{"role": "user", "content": user_message}],
)
for message in runner:
    final = message  # 툴 호출이 끝나면 마지막 메시지가 최종 응답
```

## 8. API 설계

| Method | Path | 설명 |
|--------|------|------|
| POST | `/chat` | 튜터와 대화 (user_id, message → 에이전트 응답) |
| GET | `/users/{id}/memory` | 오답노트·약점 요약 조회 |
| POST | `/quiz/answer` | 퀴즈 답안 제출·채점 |
| GET | `/stats` | 총 요청 수, 평균 지연, 누적 토큰·추정 비용 |
| GET | `/health` | 헬스체크 |

5개면 충분하다. 이 프로젝트의 무게 중심은 API 개수가 아니라 에이전트·평가 파이프라인이다.

## 9. 평가 파이프라인 (이 프로젝트의 차별점)

공고가 명시한 "자동화된 평가 파이프라인을 통해 응답 품질과 행동을 검증"의 축소 구현.

- **시나리오 셋**: 대표 질문 30개 (문법 질문 15, 오답 유도 상황 10, 탈옥 시도 — "그냥 정답 알려줘" 5)
- **평가 기준 3개** (LLM-as-judge, 기준당 1~5점):
  1. **행동 준수**: 정답을 즉시 노출하지 않고 유도했는가
  2. **사실 정확성**: 문법 설명이 검색된 노트와 일치하는가
  3. **교육적 품질**: 힌트가 학습자 수준에 맞고 다음 단계로 이어지는가
- **실행**: `python -m eval.run --prompt-version v3` → `eval/reports/v3.md` 점수 리포트 생성
- **개선 루프의 증거 남기기**: 프롬프트 v1 → v2 → v3 점수 변화를 README 표로 기록. "v2에서 탈옥 방어 점수가 낮아 규칙을 추가했더니 4.1 → 4.7" 같은 서사가 면접에서 가장 강력한 재료다.
- 비용 절감: 평가 배치는 Batches API(50% 할인)로 돌리는 것을 고려 — README에 한 줄 기록

## 11. 일정 계획 (4주)

| 주차 | 목표 | 산출물 |
|------|------|--------|
| 1주차 | 기초 학습 + 뼈대 | Claude API·tool use 학습, FastAPI 셋업, RAG 없는 튜터 챗 v1 + Streamlit UI |
| 2주차 | RAG + 에이전트화 | 문법 노트 작성 → ChromaDB 구축, 툴 4개 정의, tool runner 루프 완성 |
| 3주차 | 메모리 + 운영 | 오답노트 기록/주입, 퀴즈 툴, 토큰·지연·비용 로깅 + `/stats` |
| 4주차 | 평가 + 마무리 | 평가 파이프라인, 프롬프트 2~3회 개선 사이클, 테스트·CI, README·트러블슈팅 정리 |

배포는 필수가 아니다 — LLM API 키·비용 문제로 공개 배포 대신 **데모 영상(1~2분) + 로컬 실행 가이드**로 대체해도 충분하고, 그 이유를 README에 적으면 오히려 비용 감각 어필이 된다.
